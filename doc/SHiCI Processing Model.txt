SHiCI Processing Model


------

Purpose（目的）

SHiCI Processing Modelは、

SHiCIがどのように世界を理解し、
知識を構築し、
人間へ届けるのかを定義する。

この文書はEngineの構成を定義するものではない。

SHiCIという知識基盤の思考プロセスを定義する文書である。

------

Overview

SHiCIは次の六段階で知識を構築する。

Input
    ↓
Understand
    ↓
Resolve
    ↓
Compose
    ↓
Express
    ↓
Output

各段階は独立した責務を持ち、

前段階の結果を受け取り、

次の段階へ受け渡す。

------

1. Input

外部から情報を受け取る。

入力元は限定しない。

例えば、

* チャット
* API
* CSV
* センサー
* 他システム

など、

すべてInputとして扱う。

Inputでは意味を解釈しない。

受け取ることだけが責務である。

------

2. Understand

Inputの意味を理解する。

ここでは、

ユーザーが何を伝えたいのかを理解する。

例えば、

「ワンワン」

という入力があった場合、

* 製品名なのか
* 顧客名なのか
* 金型名なのか

まだ決定しない。

Understandでは、

入力の意図を理解する。

------

3. Resolve

理解した内容を、

SHiCIの世界へ対応付ける。

ここで初めて、

Entity

Relation

Context

などが確定する。

曖昧さが残る場合は、

追加の情報を求める。

Resolveは、

現実世界とSHiCIの世界を接続する役割を持つ。

------

4. Compose

Resolveされた情報を基に、

知識を構築する。

Composeでは、

Entityだけを見るのではない。

必要に応じて、

* Snapshot
* Relation
* Event
* History
* Context

を組み合わせ、

目的に応じたKnowledgeを構成する。

SHiCIにおける知識は、

Composeによって形成される。

------

5. Express

Knowledgeを、

人間または外部システムが理解できる形へ変換する。

表現方法は限定しない。

例えば、

* 自然言語
* Entity
* Timeline
* Graph
* Table
* JSON

など、

利用者に最適な形式で表現する。

Expressは、

Knowledgeを意味のある表現へ変換する責務を持つ。

------

6. Output

Expressされた結果を、

外部へ届ける。

Outputは、

意味を変更しない。

届けることだけを責務とする。

出力先は限定しない。

例えば、

* Web画面
* API
* AI
* 他システム

など、

すべてOutputとして扱う。

------

Processing Principles

各段階は、

一つの責務だけを持つ。

一つの段階が、

複数の責務を持ってはならない。

Processingは常に、

InputからOutputへ一方向に流れる。

途中の結果を書き換えるのではなく、

次の段階へ受け渡す。

KnowledgeはComposeで構築される。

表現はExpressで行う。

両者を混同してはならない。

------

SHiCI Principle

SHiCIは答えを探すシステムではない。

世界を理解し、知識を構築し、最適な形で表現するシステムである。
